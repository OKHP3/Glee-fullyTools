## 🔴 CRITICAL — Fix before next public share

---

### [C-01] Homepage nav logo link is broken
**Page:** Homepage (`/`)
**Finding:** The nav logo `<a>` tag wraps the butterfly image but its `href` is empty (`<>`). Clicking the logo on the homepage does nothing — or navigates to a literal empty string, which is a broken link by any validator standard.
**Expected:** `href="https://glee-fully.tools/"` or `href="/"`
**Impact:** Every visitor who clicks the logo on the homepage hits a dead link. This is the most common nav interaction on mobile.

---

### [C-02] `meta-color-scheme` still inconsistent across pages
**Pages affected:** Persona (`dark light`), About (`light` only)
**Finding:** The May 12 audit standardized most pages to `light dark` but two pages still have non-canonical values:
- `/persona/` = `dark light` (reversed preference — signals dark is the default)
- `/about/` = `light` (no dark mode support at all)
**Expected:** `light dark` on every page without exception
**Impact:** Persona page will appear in dark mode by default on OS-set-to-dark devices. About page will never dark-mode. Inconsistent user experience.

---

### [C-03] About page title and og:title mismatch
**Page:** `/about/`
**Finding:**
- `<title>`: `About — Glee-fully Personalizable Tools™ 🧰🌳`
- `og:title`: `About Glee & Jamie — Glee-fully Personalizable Tools™`
These must either match or the discrepancy must be intentional and documented.
**Impact:** Social share cards show a different title than the browser tab. Confusing for anyone sharing the About page.

---

### [C-04] Branch pages show under-construction overlay BEFORE nav in DOM
**Pages affected:** All 7 Tool branches (`/toolbox/01-discovered-careers/` confirmed, assumed others)
**Finding:** The under-construction notice (image + dismiss button) appears in the HTML source BEFORE the `<nav>`. This means on slow connections or with JS disabled, the overlay renders first and nav is inaccessible until below-fold scroll or JS fires.
**Expected:** Nav must be the first structural element inside `<body>` after the skip link. The overlay should be injected after nav, or positioned via CSS `position: fixed` so it doesn't displace the document flow.
**Impact:** Accessibility failure, bad experience on slow connections, nav temporarily unreachable.

---

### [C-05] Neighborly Bazaar missing `viewport-fit=cover`
**Page:** `/toolbox/05-organized-life/05f-neighborly-bazaar/`
**Finding:** Viewport tag is `width=device-width, initial-scale=1` — missing `viewport-fit=cover`
**Expected:** `width=device-width, initial-scale=1, viewport-fit=cover` (consistent with homepage and most other pages)
**Impact:** On notched iPhones (iPhone X and later), the status bar area may not be filled with the theme color, creating a visual gap at the top.

### [H-01] "Today's Sparkle" is permanently hardcoded to Neighborly Bazaar
**Pages affected:** All pages where banner appears
**Finding:** Every single page with a Sparkle banner shows the identical Neighborly Bazaar entry. This was the newest Tool-ette at launch but is now stale. There is no rotation mechanism or easy single-file update path.
**Recommendation:** Create `assets/data/sparkle.json` with current spotlight data. Use a small inline script to populate the banner from that file. When a new Tool-ette launches, update one JSON file instead of 60 HTML files.
**Impact:** Banner loses credibility as a "fresh pick" signal if it never changes. Users who visit twice see the same spotlight.

---

### [H-02] Skip link text inconsistency
**Pages affected:** Ecosystem (`/ecosystem/`) vs others
**Finding:** Homepage, About, Contact, Legal, and most pages use `Skip to main content` with anchor `#main`. The Ecosystem page uses `Skip to content` with anchor `#main`. Minor but flags as inconsistent to accessibility validators.
**Expected:** `Skip to main content` on every page, anchoring to `<main id="main">`.

---

### [H-03] About dropdown in nav duplicates parent link
**Pages affected:** All pages (nav is global)
**Finding:** The nav "About" parent item and the first dropdown item "About Us" both resolve to `/about/`. This creates a redundant link pair and is a usability smell — users expect the dropdown parent to either be non-clickable or to link somewhere different from its children.
**Recommendation:** Either make "About" the nav parent non-clickable (`href="#"` with `aria-haspopup` only) or remove "About Us" from the dropdown since the parent already covers it. The current structure means the `/about/` page is linked twice from the same nav element.

---

### [H-04] `meta-keywords` and `meta-revisit-after` are deprecated tags
**Pages affected:** Homepage (confirmed), several others likely
**Finding:** Both tags are ignored by Google, Bing, and all major crawlers:
- `meta-keywords` has been irrelevant since ~2009
- `meta-revisit-after` was never a standard and no major engine honors it
**Impact:** No direct harm but adds dead weight to every `<head>` block. The presence of `meta-keywords` can also signal to some spam filters that a page is using old black-hat techniques.
**Recommendation:** Remove both tags from all 60 pages via script.

---

### [H-05] Hero image has generic alt text
**Pages affected:** Homepage, About, Toolbox hub (all use same `ButterflyLoopLeft Wide 1536.png`)
**Finding:** Alt text is `Glee-fully hero illustration` — functional but generic. Screen reader users hear no meaningful description of what the image conveys.
**Expected example:** `"The Glee-fully butterfly perched on a blooming tree filled with colorful toolbox branches — a visual metaphor for the suite's seven life-organization tools"`
**Impact:** WCAG 2.2 AA requires meaningful, context-appropriate alt text on all non-decorative images.

---

### [H-06] OG image is the same generic butterfly across all pages
**Pages affected:** Homepage, About, Persona, Contact, Legal, Universe, and most Tool-ette pages
**Finding:** `og:image` defaults to the butterfly logo (`ButterflyWaiting Square 1024.png`) on most pages. Tool branch pages correctly use their branch GPT icon. But supporting pages (Contact, Legal, Persona, About) have no page-specific OG image.
**Impact:** All social shares from non-branch pages look identical — same card thumbnail, no visual differentiation.
**Recommendation:** Persona page should use a persona-specific image. About should use a "Glee & Jamie" image. Contact can remain as butterfly — it's a utility page.

### [H-07] Footer nav order doesn't match main nav order

Footer nav order: Why Glee-fully → Ecosystem → Voice & Muse → Universe → About → Contact → Legal

### [M-01] Hero images are unoptimized for mobile
**Pages affected:** Homepage, About, Toolbox hub (all use 1536px-wide PNGs)
**Finding:** The butterfly hero image (`ButterflyLoopLeft Wide 1536.png`) is served at full 1536px width regardless of viewport. A mobile visitor at 390px wide downloads a 1536px asset. No `<picture>` element, no `srcset`, no WebP variant.
**Recommendation:** Add responsive image treatment:
```html
<picture>
  <source srcset="/assets/img/hero-butterfly-768.webp" type="image/webp" media="(max-width: 768px)">
  <source srcset="/assets/img/hero-butterfly-1536.webp" type="image/webp">
  <img src="/assets/img/Glee-fullyTools ButterflyLoopLeft Wide 1536.png" alt="..." loading="eager" width="1536" height="640">
</picture>
```
**Impact:** Significant LCP improvement on mobile. Core Web Vitals score.

### [M-02] GPT icons served at 1024px, displayed at ~200-400px
**Pages affected:** All Tool-ette pages, homepage "New from the Toolbox" section
**Finding:** GPT icons are 1024x1024 PNG files. They're displayed at much smaller sizes (200-400px typically). No `srcset`, no resized variants, no WebP alternatives.
**Impact:** Mobile users download ~200-400KB per icon to display it at 1/4 the resolution. Contributes to slow LCP on Tool-ette pages.

### [M-03] Branch pages (01-07) all show under-construction but are fully navigable
**Pages affected:** All 7 branch pages
**Finding:** Each branch page opens with a full-width under-construction banner. The dismiss button works, but first-time visitors see no content on arrival. The branch pages do contain full content below the banner.
**Impact:** High bounce risk. Users who don't know to dismiss the banner may leave thinking the page is broken. The "I understand — let me explore" CTA text is clear but the visual dominance of the banner creates friction.
**Recommendation:** Either graduate these pages out of under-construction mode or reduce the banner to a slim inline notice rather than a full-page interrupt.

## FIX 6 — Remove deprecated `meta-keywords` and `meta-revisit-after` tags [HIGH]

Write a Python script `tools/remove-deprecated-meta.py`:

```python
#!/usr/bin/env python3
"""
remove-deprecated-meta.py
Removes deprecated meta tags ignored by all major search engines:
- meta name="keywords"
- meta name="revisit-after"
Idempotent. Run from repo root.
"""
import re
from pathlib import Path

DEPRECATED_PATTERNS = [
    r'<meta\s+name=["\']keywords["\']\s+content=["\'][^"\']*["\']\s*/?>',
    r'<meta\s+name=["\']revisit-after["\']\s+content=["\'][^"\']*["\']\s*/?>',
    # Also handle reversed attribute order
    r'<meta\s+content=["\'][^"\']*["\']\s+name=["\']keywords["\']\s*/?>',
    r'<meta\s+content=["\'][^"\']*["\']\s+name=["\']revisit-after["\']\s*/?>',
]

pages = [p for p in Path('.').rglob('*.html')
         if not any(x in str(p) for x in ['assets/','attached_assets/','tools/'])]

total_removed = 0
for page in sorted(pages):
    content = page.read_text(encoding='utf-8', errors='replace')
    original = content
    for pattern in DEPRECATED_PATTERNS:
        content = re.sub(pattern, '', content, flags=re.IGNORECASE)
    # Clean up blank lines left behind
    content = re.sub(r'\n\s*\n\s*\n', '\n\n', content)
    if content != original:
        page.write_text(content, encoding='utf-8')
        total_removed += 1
        print(f"Cleaned: {page}")

print(f"\nTotal pages cleaned: {total_removed} of {len(pages)}")
```

```bash
python3 tools/remove-deprecated-meta.py
```

#!/usr/bin/env python3
"""
add-toolbox-to-footer.py
Adds /toolbox/ link to footer nav after the #why link.
Idempotent — skips pages already containing the toolbox footer link.
"""
import re
from pathlib import Path

ANCHOR = '<a href="https://glee-fully.tools/#why">Why Glee&#8209;fully</a>'
INSERT_AFTER = '<li><a href="https://glee-fully.tools/#why">Why Glee&#8209;fully</a></li>'
TOOLBOX_ITEM = '<li><a href="https://glee-fully.tools/toolbox/">Opening the Toolbox</a></li>'

pages = [p for p in Path('.').rglob('*.html')
         if not any(x in str(p) for x in ['assets/','attached_assets/','tools/'])]

for page in sorted(pages):
    content = page.read_text(encoding='utf-8', errors='replace')
    # Skip if already has toolbox in footer context
    # Detect footer section and check within it
    footer_match = re.search(r'<footer.*?</footer>', content, re.DOTALL)
    if footer_match:
        footer_html = footer_match.group(0)
        if '/toolbox/' in footer_html:
            continue  # already there
        # Find the Why Glee-fully item in the footer and insert after it
        new_footer = re.sub(
            r'(<li><a href=["\']https://glee-fully\.tools/#why["\'][^>]*>Why Glee[^<]*</a></li>)',
            r'\1\n              ' + TOOLBOX_ITEM,
            footer_html,
            count=1
        )
        if new_footer != footer_html:
            content = content.replace(footer_html, new_footer)
            page.write_text(content, encoding='utf-8')
            print(f"Updated: {page}")

## FIX 8 — Implement `sparkle.json` for single-source-of-truth Sparkle banner [HIGH]

**Create:** `assets/data/sparkle.json`

```json
{
  "emoji": "🔦",
  "label": "Neighborly Bazaar GPT",
  "description": "helps you evaluate, price, and re-home household items with Glee",
  "suffix": "💫",
  "url": "/toolbox/05-organized-life/05f-neighborly-bazaar/",
  "updated": "2026-05-26"

```

**Create:** `assets/js/sparkle-loader.js`

```javascript
/**
 * sparkle-loader.js
 * Populates the "Today's Sparkle" banner from assets/data/sparkle.json
 * Falls back to static content if fetch fails (offline/slow).
 */
(function () {
  'use strict';

  var SPARKLE_SELECTOR = '.sparkle-banner a, .sparkle-link, [data-sparkle-link]';
  var SPARKLE_TEXT_SELECTOR = '.sparkle-text, [data-sparkle-text]';

  document.addEventListener('DOMContentLoaded', function () {
    var link = document.querySelector(SPARKLE_SELECTOR);
    if (!link) return; // No sparkle banner on this page

    fetch('/assets/data/sparkle.json?v=' + Date.now())
      .then(function (res) { return res.json(); })
      .then(function (data) {
        var text = data.emoji + ' ' + data.label + ' \u2014 ' + data.description + ' ' + data.suffix;
        link.textContent = text;
        link.href = data.url;
      })
      .catch(function () {
        // Fail silently — static content remains visible
      });
  });
})();
```

Add `<script src="/assets/js/sparkle-loader.js" defer></script>` to all pages that carry the sparkle banner. Mark the existing static banner link with `data-sparkle-link` so the script can find it.

**To update the banner site-wide in future:** Edit only `assets/data/sparkle.json`. The JS handles distribution.
