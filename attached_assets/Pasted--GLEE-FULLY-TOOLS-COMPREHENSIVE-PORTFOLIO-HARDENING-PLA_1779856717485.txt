# PART 1 — STRATEGIC ASSESSMENT

## The Portfolio Paradox

Glee-fully has a fundamental tension that no amount of metadata fixes resolves:

**The site is supposed to prove capability. Every unfixed bug disproves it.**

An enterprise architect with 28 certifications, 13+ years of system design,
and a stated specialty in precision and protocol cannot have a broken nav
logo link survive three audit cycles. Not because it’s catastrophically
broken – it’s a one-line fix – but because a visitor evaluating the site
as a portfolio artifact will notice it, and the subtext of “I knew about
this and still didn’t fix it” is worse than not knowing.

The “pre-opening flutter” framing was a smart softening device during
active construction. It is now a liability. The site has:

- 60 validated pages
- 2,048 internal links with zero broken
- Complete JSON-LD structured data coverage
- GA4 present on all pages
- Full favicon/manifest chain
- Sitemap parity

This is not a site in flutter. This is a launched product that’s still
wearing its hard-hat sticker.

## What “Portfolio Grade” Actually Requires

A site functioning as a living portfolio of Jamie Hill’s capability must
demonstrate all of the following simultaneously and visibly:

|Capability             |Site evidence required                                 |Current gap                                            |
|-----------------------|-------------------------------------------------------|-------------------------------------------------------|
|Systems architecture   |Coherent information hierarchy, documented structure   |Exists but buried in Ecosystem page                    |
|Technical precision    |Zero known bugs, no regressions, consistent patterns   |Broken logo href. 3rd cycle.                           |
|Visual design quality  |Intentional, distinctive, production-grade aesthetic   |Retro-bright is strong; execution has inconsistencies  |
|AI/GPT product design  |Working GPT integrations with clear use-case framing   |3 dead GPT links; branch pages lack routing clarity    |
|Prompt engineering     |Evident through Tool-ette quality and voice consistency|Strong where complete; gaps on stub pages              |
|Documentation craft    |Site tells its own story clearly                       |“Under construction” undermines the finished parts     |

|Performance engineering|Core Web Vitals, asset optimization                    |Never Lighthouse-tested against live site              |
|Content architecture   |Findable, navigable, self-explanatory                  |Search works; “Keep exploring” missing; dead ends exist|

## DOMAIN B — VISUAL DESIGN QUALITY

### B-01 | Retro-bright aesthetic is concept-strong but execution-inconsistent

The design direction (Nunito + Lato, rust + gold, paper backgrounds, retro card
shapes) is intentional and appropriate. The inconsistency is in execution:

- Card border-radius is not standardized across branch/toolette/supporting pages
- Button styles vary between pages (some have thick borders, some thin)
- Section dividers use different approaches (diagonal CSS vs. flat vs. gradient)
- The gold accent appears on some pages and not others without clear hierarchy logic

### B-02 | Hero section layout is the same on every page

Every page uses the same two-column split: left text + right image. This is safe
but not memorable. A portfolio-grade site uses layout variation to signal page type
– the homepage feels different from a Tool-ette page, which feels different from
the About page. Currently they all feel like variations of the same template.

### B-03 | GPT icons are 1024px squares in retro-stripe style – impressive

The branded GPT icon set is legitimately portfolio-worthy. They’re cohesive,
distinctive, and demonstrate real design system thinking. The problem: they’re
underutilized. Branch pages don’t prominently feature their branch icon.
Tool-ette pages use them but at sizes that don’t give them the presence they’ve earned.

### B-04 | Dark mode is declared but never designed for

`color-scheme: light dark` is set globally but no dark mode CSS overrides exist
beyond `--color-dark-surface` and `--color-dark-bg` variables that may or may not
be applied. A user on a dark-mode device gets either a system-inverted mess or
an unchanged light experience. Neither is intended. Neither is documented.

### B-05 | Typography hierarchy is inconsistent across pages

- Some pages use the eyebrow label pattern (`OKHP³™ UNIVERSE` in teal caps)
- Some pages don’t
- H1 sizes vary across page types without clear semantic reason
- Body line-height and paragraph spacing differ between pages

### B-06 | Motion and animation are underused

The butterfly logo animation (`ButterflyLoopLeft`) is the only motion on the site.
A “retro-bright” aesthetic implies energy and delight – subtle CSS animations on
card hover, a gentle fade-in on hero elements, a spinner or pulse on the Sparkle
banner, would all reinforce the brand personality without performance cost.

### B-07 | No print stylesheet

As a portfolio reference, people share links and sometimes print or PDF pages.
No `@media print` styles means the site prints with nav, footer, and colorful
backgrounds consuming ink.

## DOMAIN C — CONTENT QUALITY AND PORTFOLIO NARRATIVE

### C-01 | “Pre-opening flutter” framing is no longer accurate and actively undermines trust

Seven branch pages open with construction banners. The site has:

- A complete sitemap
- 60 validated pages
- Live GA4
- Published GPT links (on 39+ Tool-ettes)
- A Ko-fi support link

This is a launched product. Calling it “pre-opening” when someone is reading
a fully-formed Tool-ette page with complete copy, working icons, and a live
GPT link is cognitively dissonant. The framing is honest for the stub pages
and actively damaging for the complete ones.

**Recommended decision:** Differentiate. Complete pages (working GPT + full copy)
get a small “Live” badge in the corner. Stub pages get a minimal “Coming soon”
inline notice. The heavy construction banner retires from all 7 branch pages.

### C-02 | The About page doesn’t tell the portfolio story

`/about/` tells the human story of Glee and Jamie’s origin narrative warmly.
It does not tell the professional story. A visitor evaluating the site as a
portfolio artifact learns nothing about:

- Jamie’s background as an enterprise architect
- What OKHP³™ is and means
- What the scale of this project represents (60 pages, 42 Tool-ettes, 40+ GPTs)
- Why this specific AI product architecture is notable

The about page needs a second mode: a “What this project demonstrates” section
that functions as a portfolio statement without breaking the Glee/Jamie brand voice.

### C-03 | No “meta-portfolio” view exists

There’s no single page that answers: “What does the existence of this site prove?”
A recruiter, collaborator, or potential client landing here has to reverse-engineer
the answer from navigation. A `/portfolio/` or `/architecture/` page, or a section
within `/about/`, should make the case explicitly:

- “This site is a demonstration of: information architecture, custom GPT design,
  prompt engineering at scale, design system thinking, structured data governance,
  and production web deployment without a framework.”

### C-04 | Tool-ette pages are informationally complete but emotionally thin in places

The Neighborly Bazaar page is the fullest example of the format working well.
Some Tool-ette pages (especially the stub-adjacent ones) have thinner copy that
doesn’t convey the full warmth of the Glee voice. A portfolio visitor reading
thin Tool-ette copy will underestimate the craft that went into the fuller ones.

### C-05 | Branch pages don’t route visitors – they describe categories

When a visitor lands on `/toolbox/01-discovered-careers/`, they see category
explanation but no visual path to the specific Tool-ette they need. There’s a list
of Tool-ette names but no cards with icons, use-case summaries, or “try this if
you need X” guidance. The branch pages are indexes, not routing hubs.

### C-06 | No cross-linking between Tool-ettes

A user reading about Neighborly Bazaar (pricing and listing household items) would
benefit from knowing about related Tool-ettes in the Organized Life branch. No
“you might also like” or “related tools” section exists anywhere.

### C-07 | The Toolbox hub page undersells the product architecture

`/toolbox/` explains the trunk/branch/twig model – good. But the architecture
itself is the portfolio showcase. The information hierarchy of 7 branches, 42+
Tool-ettes, and a suite-aware routing GPT is genuinely impressive work. The page
buries this in prose rather than making the architecture visible as a designed system.

## DOMAIN D — PERFORMANCE AND TECHNICAL HEALTH

### D-01 | No Lighthouse baseline exists

Three audits have been run against this site. Not one included a Lighthouse score.
For a portfolio site, the Performance, Accessibility, Best Practices, and SEO
scores from Lighthouse are the four numbers any developer evaluating the site will
check first. Unknown scores are as damaging as bad scores.

### D-02 | Hero images are unoptimized (confirmed)

`ButterflyLoopLeft Wide 1536.png` is served at 1536px width to every viewport.
No WebP, no `srcset`, no `<picture>`. Mobile visitors download a desktop asset.
This is a direct LCP hit.

### D-03 | GPT icons (1024×1024 PNG) have no resized variants

205 PNG assets. No WebP. No responsive variants. Displayed at 200-400px.
Every Tool-ette page load includes at minimum one unnecessary 200-400KB image.

### D-04 | Mermaid.js loads on pages that don’t need it (unconfirmed)

If the Mermaid CDN script is in the shared `app.js` or loaded globally, every
page that doesn’t use a diagram still downloads Mermaid’s full bundle (~170KB min).
This needs verification and surgical loading.

### D-05 | Search index is 132KB raw / 30KB gzipped – verify lazy load

The search index is served only when the search modal opens (correct behavior if
implemented). Verify the fetch happens on modal open, not on page load.

### D-06 | CSS is 105KB – audit for dead rules

After many iterative fixes, the CSS likely contains orphaned rules for removed
components. PurgeCSS or a manual audit should identify and remove unused selectors.

### D-07 | No `<link rel="preload">` for LCP images

The butterfly hero is the LCP element on most pages. No preload hint means the
browser discovers it only after parsing the HTML, delaying the largest contentful
paint. One line in `<head>` per page would address this.

### D-08 | Core Web Vitals are unknown

LCP target: under 2.5s. CLS target: under 0.1. INP target: under 200ms.
None have been measured against the live site. A portfolio demonstrating technical
depth cannot have unknown performance metrics.

## DOMAIN E — ACCESSIBILITY

### E-01 | Keyboard navigation has never been browser-tested

Source-level checks confirm skip links, landmarks, and alt text exist. They do not
confirm that keyboard focus is visually apparent, that the dropdown nav is
operable by keyboard, that the search modal traps focus correctly, or that the
Mermaid diagram nodes are reachable by keyboard.

### E-02 | Color contrast on brand tokens has never been formally checked

- Rust `#d35b2d` on Paper `#f6f2ee`: requires calculation (likely passes for large text, may fail for small)
- Gold `#f3b932` on Espresso `#2a2320`: requires calculation
- The Today’s Sparkle banner (gold background + espresso text) needs WCAG 2.2 contrast audit

### E-03 | Touch target sizes on mobile unconfirmed

WCAG 2.2 Success Criterion 2.5.8 (Minimum: AA) requires touch targets of at
least 24×24 CSS pixels. The nav items, footer links, and small CTA buttons need
measurement at 390px viewport.

### E-04 | Focus visible on custom components

Bootstrap’s default focus styles may be overridden by the custom CSS. Any
`outline: none` or `outline: 0` in `theme.css` is a WCAG 2.4.11 violation.

### E-05 | Reduced motion preference

`@media (prefers-reduced-motion: reduce)` must suppress the butterfly loop
animation and any other CSS animations. Confirm this is implemented.

### E-06 | No language declaration on HTML element (needs confirmation)

`lang="en"` on `<html>` was identified as missing on some pages in earlier audits.
Confirm all 60 pages have this attribute.

## DOMAIN H — GOVERNANCE AND REGRESSION PREVENTION

### H-01 | The same bugs survive multiple audit cycles

The nav logo `href="<>"` was in the live site today. The `meta-keywords` tag was
in the live site today. The `persona` color-scheme regression was in the live site
today. Three prior audit cycles. None of these are hard fixes. They survive because
there is no pre-deploy validation gate – only post-deploy audit.

**Root cause:** The workflow is:
`Replit fixes → commit → push → live site`
with no automated check between “fixed in Replit” and “live on GitHub Pages.”

**Required:** A GitHub Actions workflow that runs `validate-site.py` and
`check-links.py` on every push to `main` and blocks the merge if they fail.
This converts the existing audit scripts from retrospective tools into
prospective gates.

### H-02 | No version of the site is “locked as baseline”

There’s no tagged release, no `CHANGELOG.md`, no way to know what changed
between the May 2 audit and the May 26 evaluation. Changes that introduce
regressions (like the persona color-scheme not getting committed) are invisible.

### H-03 | `assets/templates/` may not have been created

The template library task was specified in a prior session. Confirm it exists.
If not, the next time a new Tool-ette page is needed, the process starts from
scratch instead of from a validated template.

### H-04 | `sparkle.json` dynamic banner was specified but not confirmed deployed

The single-source sparkle architecture was designed in the evaluation prompt.
The live site still shows the same hardcoded Neighborly Bazaar banner.
If `sparkle.json` doesn’t exist, updating the banner still requires touching 60 files.

  """Remove deprecated meta tags from all HTML pages. Idempotent."""
  import re
  from pathlib import Path

  PATTERNS = [
      r'\s*<meta\s+name=["\']keywords["\']\s+content=["\'][^"\']*["\']\s*/?>',
      r'\s*<meta\s+name=["\']revisit-after["\']\s+content=["\'][^"\']*["\']\s*/?>',

  ]

  pages = [p for p in Path('.').rglob('*.html')
           if not any(x in str(p) for x in
                      ['assets/','attached_assets/','tools/'])]

  cleaned = 0
  for page in sorted(pages):
      text = page.read_text(encoding='utf-8', errors='replace')
      original = text
      for pat in PATTERNS:
          text = re.sub(pat, '', text, flags=re.IGNORECASE)

      if text != original:
          page.write_text(text, encoding='utf-8')
          cleaned += 1
          print(f"Cleaned: {page}")
  print(f"Done. {cleaned} pages cleaned.")
  PYEOF

  EOF

── 6B: Create sparkle-loader.js ─────────────────────────────────────────

  cat > assets/js/sparkle-loader.js << 'EOF'
  (function () {
    'use strict';
    document.addEventListener('DOMContentLoaded', function () {
      var link = document.querySelector('[data-sparkle-link]');
      if (!link) return;
      fetch('/assets/data/sparkle.json?v=' + Date.now())
        .then(function (r) { return r.json(); })
        .then(function (d) {
          link.textContent = d.emoji + ' ' + d.label + ' \u2014 ' + d.description + ' ' + d.suffix;
          link.href = d.url;
        })
        .catch(function () { /* fall through to static content */ });
    });
  })();
  EOF

── 6C: Add data-sparkle-link to all pages with the banner ───────────────

Find all sparkle banner links:
  grep -rn 'Sparkle\|sparkle' --include="*.html" . | grep -v assets/ | head -10

  <!-- Before -->
  <a href="/toolbox/05-organized-life/05f-neighborly-bazaar/">🔦 Neighborly...</a>
  <!-- After -->
  <a data-sparkle-link href="/toolbox/05-organized-life/05f-neighborly-bazaar/">🔦 Neighborly...</a>

Then add the script tag before </body> on those pages:
  <script src="/assets/js/sparkle-loader.js" defer></script>

PHASE 8 — ACCESSIBILITY AUDIT

Run axe accessibility checks via Playwright:

  python3 - << 'EOF'
  from playwright.sync_api import sync_playwright
  import json

  PAGES_TO_TEST = ["/", "/toolbox/", "/ecosystem/", "/about/",
                   "/toolbox/05-organized-life/05f-neighborly-bazaar/"]
  BASE = "https://glee-fully.tools"

  with sync_playwright() as p:
      browser = p.chromium.launch()
      page = browser.new_page()
      for path in PAGES_TO_TEST:
          page.goto(BASE + path, wait_until="networkidle")
          # Inject axe-core
          page.add_script_tag(url="https://cdnjs.cloudflare.com/ajax/libs/axe-core/4.9.1/axe.min.js")
          results = page.evaluate("axe.run()")
          violations = results.get("violations", [])
          if violations:
              print(f"\n{path} — {len(violations)} violation(s):")
              for v in violations:
                  print(f"  [{v['impact'].upper()}] {v['id']}: {v['description']}")
                  for node in v.get("nodes", [])[:2]:
                      print(f"    → {node.get('html', '')[:100]}")
          else:
              print(f"{path} — No axe violations")
      browser.close()
  EOF

DESIGN QUALITY STANDARD — READ BEFORE WRITING ANY NEW HTML/CSS
════════════════════════════════════════

This site is a portfolio of craft. Apply the frontend-design skill standard:

- Every new component must have an intentional aesthetic point of view
- Card hover states must feel responsive and alive (150ms ease, not instant)
- Typography must follow the hierarchy: Nunito for headings, Lato for body
- Color usage must reference CSS variables — NEVER hardcoded hex values in HTML
- Spacing must feel deliberate — generous negative space OR controlled density
- Motion must be purposeful and respect prefers-reduced-motion
- No component should look like it came from a Bootstrap template starter kit
- The retro-bright aesthetic means warmth, charm, and specificity — not busyness

When in doubt: ask "Would someone who cares about craft notice this component?"
If the answer is no, it's not portfolio grade. Fix it.
