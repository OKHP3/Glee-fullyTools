Portfolio case study

Glee‑fully as a Living AI Product System

Sixty pages. Forty‑two tool‑ettes. A shared design language, a client‑side search engine, a Python CI gate, and a governance model that keeps it all coherent as it grows — built without a single npm install.

Explore the Toolbox Meet Glee & Jamie

The design challenge

Building a coherent suite of AI tools is easy when you have three tools. It gets interesting at forty‑two. Every new tool-ette has to feel like it belongs in the same universe as the first one — same voice, same visual language, same mental model for navigation — without introducing a build system, a CMS, or a dependency that could rot overnight.

The constraints were self‑imposed and intentional: static HTML only, deployable straight to GitHub Pages, readable by any browser without JavaScript, and maintainable by a team of two without a DevOps workflow. Those constraints forced clarity in every other decision.

The constraints that shaped everything

  • No build step. No webpack, Vite, or Next.js. Every page is a real HTML file.
  • No CMS. Content lives in the HTML. Idempotent Python scripts handle bulk changes.
  • No framework JS. One shared app.js — 918 lines — covers nav, search, scroll animations, and the mobile overlay.
  • No external search API. A weighted client‑side scorer on a 130 KB JSON index.
  • No broken pages. A CI gate exits non‑zero on any metadata regression.

Information architecture: trunk, branch, tool‑ette

Navigation at scale requires a model visitors can internalize in three seconds. The trunk‑branch‑tool‑ette hierarchy gives every page an address that explains itself.

🌳

The Trunk

The Toolbox hub (/toolbox/) is the single entry point for all 42 tools. Seven branch cards — each with a name, tagline, and tool count — give visitors an immediate map of the whole system without needing to click.

🌿

The Branches

Seven thematic hubs group tools by life domain: careers, treasured finds, food, travel, organization, health, and identity. Each branch page explains the domain, introduces the tools, and lists the best‑fit visitor profiles.

🍃

The Tool‑ettes

Forty‑two pages — one per Custom GPT. Each carries a hero, a functions list, a "who this is for" aside, a live GPT link, sibling cards, and a "Keep exploring" tray so the visitor is never stranded at a dead end.

Orientation at every depth

Three independent systems reinforce the mental model at every level of depth. A visitor who lands on any page — from a search engine or a shared link — always knows exactly where they are and what's nearby.

  • Breadcrumb. Home › Branch › Tool‑ette on every inner page, mirroring the JSON‑LD BreadcrumbList for Google‑rich‑results eligibility.
  • "Keep exploring" tray. Injected on all 42 tool‑ette pages via an idempotent Python script — links to parent branch, previous sibling, next sibling, Toolbox hub, and site search.
  • Site search. ⌘K / / opens a modal that searches all 60+ pages instantly, weighted by title, heading, description, and body — zero dependencies, no API key.

Why no mega‑menu?

With 42 tools across 7 branches, a flat mega‑menu would require 42 entries — overwhelming and unmaintainable. The trunk‑branch‑tool‑ette model keeps the primary nav to five items while the search modal handles power‑user navigation. The two systems are complementary, not redundant.

Technical architecture: static by conviction

Every technical decision has a legible reason. "Static" isn't a limitation — it's the feature that makes the site fast, cheap to host, and impossible to take down with a server crash.

Zero‑dependency frontend

One CSS file (theme.css, ~4,800 lines, organized into four named brand scopes). One JavaScript file (app.js, 918 lines). Google Fonts loaded from CDN with preconnect hints. No runtime frameworks. No tree‑shaking required because there's nothing to shake.

Client‑side search engine

A Python script walks every HTML page, extracts title / description / headings / body, and writes a 130 KB JSON index. The runtime — 200 lines of vanilla JS — tokenizes queries, scores fields (title × 10, headings × 5), and renders modal results in under 50 ms. No Algolia. No Lunr. No API costs.

Python CI toolchain

Seven idempotent scripts handle bulk mutations and validation. Two of them run as GitHub Actions CI gates: validate-site.py checks every page's metadata, and check-links.py validates every internal href against the filesystem and sitemap. A push with broken metadata doesn't reach production.

What the CI gate actually checks

On every push to main, validate-site.py verifies that every HTML page has: a non‑empty <title>, a meta description, a self‑referencing canonical URL, a matching og:url, the brand theme‑color, a favicon SVG link, a webmanifest link, a parseable JSON‑LD block, and exactly one <h1>. Pages that add a Mermaid diagram must include the paid‑referral credit exactly once. The gate exits non‑zero on any critical defect — no silent regressions.

Design system: one palette, four scopes

A single theme.css file serves three different brands — OverKill Hill P³, Glee‑fully, and AskJamie — without specificity wars or !important hacks. The trick is structural, not clever.

CSS scope architecture

The stylesheet is divided into four named scopes, each separated by a boxed banner comment and a body‑class selector pattern. New rules go into the correct scope based on which brand they affect — never into a global block unless they're genuinely cross‑brand.

  • GLOBAL — tokens, reset, layout, header, footer, search engine, scroll‑reveal
  • OVERKILL — default chrome for the OverKill Hill P³ site (body:not(.glee-main):not(.askjamie-main))
  • GLEE — paper‑cream backgrounds, rust‑and‑gold accents, Fredoka headings (.glee-main)
  • ASKJAMIE — mid‑century modern teal palette (.askjamie-main)

The scope map means that merging a CSS change from a sibling brand repo is a one‑line decision: changes inside GLOBAL / CROSS‑BRAND blocks get reconciled; changes inside OVERKILL / GLEE / ASKJAMIE blocks are intentional brand divergence and stay put.

The token layer

All colors, spacing, and typography are declared as CSS custom properties on :root. No hardcoded hex values in component rules — only var(--color-rust), var(--color-teal), and friends. Dark‑mode support is a single overriding :root block at the end of the GLOBAL scope — no duplication of component rules required.

The token discipline is enforced by code‑review convention: any PR adding a hardcoded hex color to a non‑root rule is a regression and gets flagged before merge.

Governance: keeping 60 pages coherent

A site with 60 pages and two maintainers needs explicit rules — not just conventions. The governance system makes the right action obvious and the wrong action hard.

agent‑skills.md

A 10‑skill operating constitution for AI agents working on the codebase. Six core constraints (static‑only, no fabricated content, brand voice, taxonomy rules, idempotent scripts, Mermaid referral invariant) are declared as non‑negotiable. Each skill has a Purpose, Checks, and Off‑limits section so any agent knows exactly what it's allowed to do before touching a file.

AUTOGEN markers

Bulk‑generated content — breadcrumbs, JSON‑LD, keep‑exploring trays, sparkle loaders — is fenced with <!-- AUTOGEN:BLOCK --> markers. Every mutator script checks for the marker before writing, making re‑runs byte‑idempotent. Humans can safely re‑run any script on any page without creating duplicates.

Template library

Nine structural HTML templates cover every page type: homepage, toolbox hub, branch hub, tool‑ette, interior single, search utility, Mermaid diagram, error, and holding. Templates are invisible to the CI gate — the assets/ tree is skipped by all validators — so they can be updated freely without affecting production pages.

Script run‑order discipline

Seven Python scripts have a defined run order that prevents dependency failures. inject-jsonld.py reads og:image to set primaryImageOfPage, so it runs after activate-icons.py. inject-breadcrumb.py reads the JSON‑LD that inject-jsonld.py wrote, so it runs last among the mutators. The run‑order is documented in replit.md and in each script's docstring.

Why Python, not Node?

The toolchain deliberately matches the runtime: static HTML manipulated by Python's html.parser, no npm dependencies, no package.json to drift out of date. A future maintainer with only Python installed can run the full CI gate without installing anything else.

What this demonstrates

Glee‑fully isn't just a product — it's a proof of methodology. Here's the craft it documents.

🗺️

Scalable information architecture

A three‑tier taxonomy that makes 60 pages feel navigable without a mega‑menu or a site map page. Every page always knows what's above it, beside it, and below it — and communicates that to both humans and search engines.

⚙️

Zero‑dependency engineering

A fully functional product site — search, dark mode, lazy loading, scroll animations, mobile nav — built without a single npm package or a build pipeline. Every dependency is a deliberate choice, not a scaffold default.

🎨

Multi‑brand design system

One stylesheet, four scopes, three brands — no specificity wars. A token‑based color system that supports dark mode and future brand extensions without touching component rules.

🔬

CI‑gated quality

Metadata regressions, broken links, stale search indices, and missing JSON‑LD don't reach production. The gate is Python, runs in GitHub Actions, and exits non‑zero on any critical defect — no exceptions.

✨

Prompt engineering at scale

40+ Custom GPTs share a personality, a canon, and a voice without blurring into each other. Consistency at that scale is an information‑architecture problem, not just a writing problem — and this project solves it structurally.

📋

Governance for longevity

An operating constitution, AUTOGEN‑marker discipline, and template library ensure the site can grow to 100+ pages without accumulating the inconsistencies that make large static sites unmaintainable. Structure is a gift to the future self.
