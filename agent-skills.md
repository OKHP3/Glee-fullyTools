# agent-skills.md — Glee-fully Site Governance

## Purpose

This file is the operating constitution for any AI agent working on the
`OKHP3/Glee-fullyTools` repository. Read it before touching any file.

Glee-fully is a static GitHub Pages site and a living portfolio artifact.
Every unfixed bug, broken link, or inconsistent style is evidence against
the portfolio claim. Every agent session must leave the site at least as
good as it found it — and preferably better.

## Core Constraints (Never Violate)

1. **Preserve static GitHub Pages architecture.** No Node.js build step, no
   webpack, no framework unless the owner explicitly approves. `python3 -m
   http.server` in dev; static files deployed from repo root.

2. **No build step without owner approval.** WebP conversion, PurgeCSS,
   bundling — all require explicit sign-off. The current site ships raw
   HTML/CSS/JS directly.

3. **Preserve Glee-fully brand voice.** Warm, human, retro-bright, approachable,
   practical. Not corporate. Not generic SaaS. Not technical jargon for non-technical
   users. Copy must read like a person who cares about craft and human connection.

4. **Preserve trunk → branch → tool-ette taxonomy.** The information architecture
   is Toolbox (trunk) → 7 branches → 42 tool-ettes. Do not flatten, rename, or
   reorder this hierarchy without explicit owner instruction.

5. **Do not fabricate GPT URLs, claims, testimonials, metrics, or legal statements.**
   If a GPT URL is unknown, use the verified placeholder pattern already in the HTML.
   Never invent tool descriptions that differ from the tool's actual purpose.

6. **Prefer idempotent scripts and AUTOGEN markers over one-off edits.** Site-wide
   changes must be done via scripts that include "skip if already present" guards.
   Scripts live in `scripts/`. AUTOGEN markers (e.g. `<!-- AUTOGEN:BREADCRUMB -->`)
   protect generated blocks from double-injection.

## CSS Scope Map

`assets/css/theme.css` is 4,700+ lines organized into brand scopes separated
by `╔══╗` boxed banners. Edit in the **correct scope only**.

| Scope | Starts at | Selector pattern | What lives here |
|---|---:|---|---|
| GLOBAL | L 6 | unscoped / `:root` / `body` | Tokens, reset, layout, nav, footer, search, Ko-fi |
| OVERKILL | L 2215 | `body:not(.glee-main):not(.askjamie-main)` | OverKill Hill default chrome |
| GLEE | L 2633 | `.glee-main …` | Glee-fully brand chrome (paper hero, breadcrumb, milestone) |
| ASKJAMIE | L 3659 | `.askjamie-main …` | AskJamie chrome |
| CROSS-BRAND | L 4116 | `.glee-main … , .askjamie-main …` | Shared Glee+AskJamie overrides |

**Rules:**
- Never use hardcoded hex values in HTML or new CSS rules. Always use `var(--color-*)` tokens.
- Never add `<style>` blocks to HTML files. All CSS belongs in `theme.css`.
- Never add `style="…"` attributes except where a pre-existing multi-property
  inline style cannot be extracted to a class without a disproportionate refactor
  (currently: 2 lines in `toolbox/06-healthy-bee-ing/index.html`).
- When diffing against sibling repos: GLOBAL/CROSS-BRAND differences should be
  reconciled; OVERKILL/GLEE/ASKJAMIE differences are intentional brand divergence.

## 10 Agent Skills

### skill.static-site-auditor

**Purpose:** Verify GitHub Pages readiness and structural integrity.

**Checks:**
- `python3 scripts/validate-site.py` exits 0 (60/60 pages, 0 issues)
- `python3 scripts/check-links.py` exits 0 (0 broken internal links, sitemap parity)
- All internal hrefs use root-absolute paths (`/assets/…`, `/toolbox/…`)
- No `<script>` tags without `defer` or `async` (no render-blocking JS)
- No page introduces a build step or external runtime dependency

**Off-limits:**
- Do not introduce `node_modules`, `package.json`, or bundler config
- Do not change `publicDir` in deployment config without owner approval

---

### skill.responsive-render-tester

**Purpose:** Confirm the site renders correctly across all device widths.

**Viewports to test:** 320, 375, 390, 414, 768, 1024, 1280, 1440 px wide.

**Checks per page:**
- No horizontal overflow (`scrollWidth ≤ clientWidth + 5px`)
- Nav is visible and usable; hamburger works on mobile
- Skip link is present and functional
- Search modal opens, closes, and does not trap focus
- Breadcrumbs wrap cleanly (do not overflow)
- Hero image does not crush text or overflow container
- CTA buttons remain tappable (≥ 44×44 px on mobile)
- Footer stacks correctly
- Mermaid diagrams do not cause horizontal overflow
- No console errors from missing assets or JS exceptions

**Tooling:** `scripts/viewport-qa.py` (Playwright), `scripts/responsive-audit.py` (static analysis).
Screenshots for failures go to `assets/audit/screenshots/`.

**Off-limits:**
- Do not add page-specific CSS hacks. All responsive fixes go in `theme.css` GLOBAL scope.

### skill.portfolio-positioning-reviewer

**Purpose:** Ensure the site functions as a credible portfolio artifact, not just a working site.

**For every major page, answer:**
1. Does it explain what it is within 5 seconds?
2. Does it explain who it helps?
3. Does it show why the system exists?
4. Does it prove the creator can design and ship?
5. Does it offer a clear next step?

**Checks:**
- Homepage hero explains value proposition immediately
- Every tool-ette has a plain-English utility sentence (not just a playful name)
- Branch pages function as visual routing hubs, not text indexes
- About page tells the creator story and makes the portfolio case
- `/showcase/` (if present) provides the case-study framing
- No page is a dead end — every page has at least one clear next action

**Off-limits:**
- Do not rewrite brand voice to sound corporate or technical
- Do not invent capability claims not backed by visible outputs

---

### skill.glee-brand-guardian

**Purpose:** Preserve the Glee-fully voice and visual identity across all agent sessions.

**Voice rules:**
- Warm, specific, proud — not corporate, not generic SaaS, not jargon-heavy
- Playful names (Personal Librarian, Menu Conductor, Dreamland Journeys) are intentional — explain utility in a subtitle, not by renaming
- Second-person, present-tense: "helps you…", "use when…", not "this tool enables users to…"
- Non-technical users must understand every page without a glossary

**Visual rules:**
- Retro-bright palette: Rust (`#d35b2d`), Gold (`#f3b932`), Espresso (`#2a2320`), Paper (`#f6f2ee`)
- Only use CSS variables: `var(--color-rust)`, `var(--color-gold)`, etc.
- Typography: Fredoka/Nunito for headings, Open Sans/DM Sans for body
- No Bootstrap utility classes, no Tailwind, no generic component library aesthetics

**Checks:**
- No hardcoded hex values in new HTML or CSS
- No `<style>` blocks or inline `style="…"` attributes added
- No generic "lorem ipsum" or placeholder copy shipped to production
- Brand spelling: always "Glee‑fully" (non-breaking hyphen `&#8209;`) in displayed text; `glee-fully` in URLs and code

### skill.accessibility-aa-reviewer

**Purpose:** Maintain WCAG 2.2 AA-oriented practical conformance.

**Checks:**
- All images have descriptive `alt` text (decorative images: `alt=""` + `aria-hidden="true"`)
- All interactive controls have accessible names (`aria-label` or visible text)
- No keyboard traps — Tab/Shift-Tab/Escape navigate freely through search modal and nav
- Focus is visible on all links, buttons, and form controls
- Sticky nav does not obscure focused elements (WCAG 2.2 2.4.11)
- Mobile tap targets ≥ 44×44 px
- Color is not the only means of conveying information
- `@media (prefers-reduced-motion: reduce)` suppresses all animations
- Mermaid diagrams have a text-based fallback or summary
- `<main id="main">` and `.skip-to-content` are on every page

**Off-limits:**
- Do not remove `aria-label` from nav toggle, search button, or breadcrumb nav
- Do not remove `role` attributes from search modal without adding equivalent semantics

---

### skill.seo-schema-curator

**Purpose:** Maintain structured data and metadata quality.

**Checks per page:**
- `<title>` is non-empty and matches the format `Page Name — Glee‑fully Personalizable Tools™ 🧰🌳`
- `<meta name="description">` is non-empty and describes the page's utility
- `<link rel="canonical">` points to the page's own URL (not the homepage)
- `og:url` matches the canonical URL
- `og:image` points to a real, resolved file
- JSON-LD blocks parse as valid JSON (run `scripts/validate-site.py`)
- `BreadcrumbList` JSON-LD matches the visible `<nav aria-label="Breadcrumb">`
- Search index is rebuilt after content changes (`scripts/build-search-index.py`)
- `sitemap.xml` includes every indexable page; `lastmod` is accurate

**Off-limits:**
- Do not set `SoftwareApplication` JSON-LD on pages that are not actual software applications
- Do not fabricate FAQ entries — only add FAQ JSON-LD where the Q/A genuinely helps visitors
- Do not change canonical URLs without updating sitemap, breadcrumb, and og:url simultaneously

### skill.conversion-path-auditor

**Purpose:** Ensure every visitor has a clear next action.

**Visitor types and their CTAs:**

| Visitor | Question | CTA |
|---|---|---|
| Casual user | "What can I use?" | Start with the Toolbox |
| AI/product peer | "How was this designed?" | View Showcase |
| Potential collaborator | "Can Jamie build systems like this?" | Contact |
| Supporter | "How do I support this?" | Ko-fi |
| Returning user | "Where was that one tool?" | Search |

**Checks:**
- Every tool-ette page has a "Keep exploring" tray (parent branch, siblings, Toolbox, Search)
- No page ends without at least one CTA link
- All GPT "Open Tool" links resolve to live ChatGPT URLs (no placeholders in production HTML)
- Ko-fi link is present in the footer on every page
- Contact email and social links are present and correct

**Off-limits:**
- Do not add more than 5 CTA links per page — CTA overload defeats conversion
- Do not add Ko-fi inline widgets in the body; footer placement is the intentional pattern

---

### skill.artifact-proof-checker

**Purpose:** Ensure portfolio claims are backed by visible evidence.

**Checks:**
- Audit reports exist and are dated correctly (`assets/docs/`)
- Validation JSON outputs exist and are recent (`assets/audit/`)
- Search index exists and was rebuilt after the most recent content change
- `sitemap.xml` and `feed.xml` are up to date
- Screenshots of responsive tests exist for any viewports that had issues
- If the site claims "60 pages" or "42 tool-ettes," count them and confirm

**Off-limits:**
- Do not delete audit reports without owner approval
- Do not update "Recent Changes" in `replit.md` with unverified claims

---

### skill.staleness-drift-detector

**Purpose:** Catch content that has become outdated or inconsistent.

**Checks:**
- No `<!-- TODO -->` comments remain in production HTML (check `OPEN_TODOS_2026-05-03.md`)
- Audit report dates match the actual dates work was performed
- `sitemap.xml` `lastmod` dates are accurate for modified pages
- Search index `count` matches the actual number of indexable pages
- `replit.md` "Recent Changes" reflects the current state
- Branch page Tool-ette lists match the actual tool-ettes present on disk
- Construction banners are only on pages with genuinely incomplete content

**Off-limits:**
- Do not silently update dates to make things look newer than they are
- Do not remove "under construction" language from pages that genuinely are incomplete

---

### skill.agent-safety-governor

**Purpose:** Prevent destructive or out-of-scope changes.

**Prevents:**
- Large framework rewrites (no React, Next.js, Vite, Astro without owner approval)
- Deleting any file in `assets/img/` without documenting which files and why
- Fabricating GPT URLs, user quotes, metrics, or legal statements
- Breaking the CSS scope model (adding GLEE-scope rules to GLOBAL scope, etc.)
- Silently changing brand voice, tool names, or taxonomy structure
- Adding `<style>` blocks, `style="…"` attributes, or `!important` hacks
- Pushing commits that break `scripts/validate-site.py` or `scripts/check-links.py`
- Removing the Mermaid referral credit from `ecosystem/` or `universe/`

**Before any large change, verify:**
1. `python3 scripts/validate-site.py` exits 0 before you start
2. Your planned change is within the scope of your assigned task
3. You are not touching files outside your task's relevant files list
4. You have a way to verify the change is correct (validator, visual check, or link check)

## Script Quick Reference

```bash
# Run before AND after any change:
python3 scripts/validate-site.py      # 60/60 pages, 0 issues = clean
python3 scripts/check-links.py        # 0 broken links, sitemap parity = clean

# Run after any HTML content change:
python3 scripts/build-search-index.py # rebuilds assets/data/search-index.json

# Mutators (idempotent — safe to re-run):
python3 scripts/normalize-head.py     # canonical favicon chain + theme-color
python3 scripts/activate-icons.py     # hero img + og:image → per-tool icons
python3 scripts/inject-jsonld.py      # JSON-LD @graph + BreadcrumbList

# Asset / feed regenerators:
python3 scripts/audit-assets.py       # icon-map.json + asset inventory
python3 scripts/generate-feed.py      # feed.xml

# CI gate (runs automatically on push/PR to main):
# .github/workflows/validate.yml → validate-site.py + check-links.py + build-search-index.py
```
